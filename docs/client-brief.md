# Client Brief

## Objetivo del brief

Usa este documento para transformar la información del cliente en input claro para discovery, alcance, propuesta comercial y ejecución.

## Cómo usarlo dentro del workflow

- Completar después de la llamada de discovery.
- Convertirlo en un `project scope` aprobado.
- Usarlo como base para la propuesta y la cotización.
- Detectar temprano si el proyecto requiere admin, media management, integraciones o backend real.

## 2. Tipo de proyecto

- ¿Necesita una landing page, sitio institucional, SPA o SaaS ligero?
- ¿El proyecto empieza pequeño pero debe crecer a panel administrativo?
- ¿Habrá diferentes perfiles de usuario en el futuro?
- ¿Es un sitio para vender servicios, captar leads o soportar una operación interna?

## 3. Objetivo de negocio

- ¿Cuál es la meta principal del proyecto?
- ¿Qué acción quieres que haga el usuario?
- ¿Cómo se medirá el éxito?

## 4. Alcance funcional

- ¿Se requieren formularios? ¿Cuáles?
- ¿Se necesita blog, recursos o centro de contenido?
- ¿Habrá tienda, checkout o pagos?
- ¿Se necesita backend propio?
- ¿Se necesita base de datos?
- ¿Se necesita panel administrativo?
- ¿Qué debería poder editar el cliente por sí mismo?
- ¿Qué queda explícitamente fuera de esta primera fase?

## 5. Contenido y branding

- ¿Ya existe branding?
- ¿Hay manual de marca?
- ¿Hay copies o textos listos?
- ¿Hay estructura de secciones definida?
- ¿Quién aprueba contenido y diseño?

## 6. Diseño

- ¿Existe diseño en Figma u otra herramienta?
- ¿Hay referencias visuales obligatorias?
- ¿Qué sitios admiran o quieren evitar?
- ¿Buscan algo más sobrio, premium, técnico o agresivo comercialmente?

## 7. Multimedia

- ¿Qué imágenes necesita el proyecto?
- ¿Qué videos necesita el proyecto?
- ¿Ya tienen fotos, renders, mockups o video?
- ¿Necesitan optimización o edición de assets?
- ¿Esperan poder subir o reemplazar media desde un panel?

## 8. Integraciones

- ¿Hay CRM, email marketing, analytics o chat que deban conectarse?
- ¿Se requiere WhatsApp, HubSpot, Mailchimp, Stripe u otra herramienta?
- ¿Existen APIs o sistemas internos que deban hablar con el sitio?
- ¿Alguna integración debe quedar solo preparada, pero no implementada todavía?

## 9. Operación interna

- ¿Quién administrará el contenido?
- ¿Cuántas personas necesitarán acceso al admin?
- ¿Requieren historial de cambios, aprobación o roles?

## 10. Restricciones

- ¿Hay deadline fijo?
- ¿Hay presupuesto objetivo o rango?
- ¿Hay requerimientos legales, de privacidad o compliance?
- ¿Hay limitaciones de hosting o stack?

## 11. Entregables esperados

- ¿Qué esperan recibir exactamente?
- ¿Quieren solo frontend o también acompañamiento de despliegue?
- ¿Habrá mantenimiento posterior?
